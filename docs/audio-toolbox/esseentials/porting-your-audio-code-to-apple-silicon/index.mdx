## 概述

在您的移植计划中包括迁移使用 Core Audio 系列框架的代码的时间。特别是，更新您的音频单元以支持 Apple 芯片，并优化任何实时代码以在所有 Mac 计算机上高效运行。

创建主机应用程序和音频单元的通用版本 in page link
始终创建音频主机应用程序和音频单元插件的通用版本。通用二进制文件可确保您的代码在所有平台上本机运行，这使您有机会针对每个平台优化代码。

由于主机应用程序控制执行环境，因此提供通用音频单元尤为重要。如果您的音频单元包含 arm64 和 x86_64 架构的代码，则主机应用程序可以在进程内或进程外加载您的音频单元。对于某些应用程序来说，进程内加载音频单元的能力仍然很重要，例如那些需要最小延迟的应用程序。

要了解如何创建通用二进制文件，请参阅构建通用 macOS 二进制文件。

## 支持现代音频组件架构 in page link

对于 arm64 架构，请始终使用音频组件 API 将音频单元、编解码器和其他代码模块加载到您的应用中。音频组件 API 是搜索可加载代码模块的现代方法，可在 macOS 10.6 及更高版本中使用。面向 arm64 架构或链接到 macOS 11 SDK 的应用程序无法使用旧版 Carbon Component Manager API 来打开音频单元。如果您的应用程序使用 Carbon 组件管理器，请计划在移植应用程序时将其迁移出去。

如果您开发音频单元或编解码器，请更新您的代码以支持音频组件或音频单元扩展 API（如果您尚未这样做）。当您将音频单元或编解码器链接到 macOS 11（或更高版本）SDK 时，请使用这些现代 API 之一，而不是 Carbon 组件 API。

:::note

为了保持兼容性，链接到 macOS 10.15（或更早版本）SDK 的应用程序可能会继续使用 Carbon 组件管理器来加载音频单元和编解码器。同样，使用 macOS 10.15（或更早版本）SDK 构建的音频单元和编解码器可能会继续包含单独的资源和入口点以支持旧版 API。

:::

有关音频组件 API 的信息，请参阅音频工具箱框架。

## 使用工作组确定实时线程的优先级 in page link

执行实时音频处理的应用程序需要确保其线程定期运行。在某些情况下，您的应用程序或音频单元可能还需要与音频服务器或主机应用程序的线程协调，以确保及时处理音频。对于这两种情况，请使用工作组将实时线程的调度需求传达给系统。

每个核心音频设备都提供一个工作组，其他实时线程可以使用 os_workgroup_join_self 函数加入该工作组。加入音频设备工作组会告诉系统您的应用程序的实时线程正在努力争取与设备线程相同的截止日期。您可以通过以下几种方式之一访问与设备关联的工作组：

- 获取设备的 kAudioDevicePropertyIOThreadOSWorkgroup 属性。
- 获取音频 I/O 单元（AUHAL 或 AURemoteIO）的 kAudioOutputUnitProperty_OSWorkgroup 属性。
- 从充当输入/输出单元的 AUAudioUnit 对象的 osWorkgroup 属性获取它。

如果您的音频单元创建自己的实时音频处理线程用于渲染，请通过将线程加入主机应用程序的工作组来协调这些线程与主机应用程序的活动。要获取主机应用程序的工作组，请从 AUAudioUnit 对象的 renderContextObserver 属性返回 AURenderContextObserver 块。 （对于 v2 音频单元，返回 AURenderContextObserver 块作为音频单元 kAudioUnitProperty_RenderContextObserver 属性的数据。）当系统执行您的块时，从提供的 AudioUnitRenderContext 对象。主机的工作组可能会在渲染调用之间发生变化。如果是这样，请更新您的实时音频线程以加入系统传递给您的块的工作组，并离开其之前的工作组。

如果您的实时音频线程在与核心音频线程不同的截止日期上运行，请使用 AudioWorkIntervalCreate 函数创建您自己的间隔工作组。对于您创建的工作组，将线程加入工作组并从一个线程调用 os_workgroup_interval_start 函数。当您调用该函数时，请指定您期望线程唤醒并处理音频的时间。完成工作后，调用 os_workgroup_interval_finish 函数告诉系统您已完成与当前截止日期相关的工作。

有关 Core Audio 工作组的信息，请参阅 Audio Toolbox 的参考。
